# AGENTS.md

This file provides guidance to agents when working with code in this repository (Architect mode).

## 关键架构约束与设计决策（非显而易见）

### 1. 串口通信的固定帧长设计
- **设计决策**: 采用固定 11 字节帧格式而非变长帧
- **原因**: 简化 DMA 接收逻辑，避免复杂的状态机
- **约束**: 所有数据必须适配 6 字节数据区，16-bit 数据采用高低字节拆分
- **影响**: 新增协议 ID 时必须遵守这个约束，否则需要重新设计整个通信栈

### 2. ProtocolFrame_t 的 12 字节对齐设计
- **设计决策**: 结构体定义为 12 字节（含 `reserved` 补齐位），但只发送 11 字节
- **原因**: 确保内存对齐，避免编译器自动填充导致的不可预测行为
- **约束**: TX 任务必须显式指定发送 11 字节，不能使用 `sizeof(txFrame)`
- **影响**: 任何修改 `ProtocolFrame_t` 结构体的操作都必须同时更新发送长度

### 3. DMA 接收的重启机制
- **设计决策**: 每次 IDLE 中断后重启 DMA，而非连续接收
- **原因**: 重置 DMA 计数器 (NDTR)，确保每帧数据的完整性
- **约束**: 必须在 `HAL_UARTEx_RxEventCallback()` 中调用 `HAL_UART_AbortReceive()` 然后重新初始化
- **影响**: 接收延迟增加，但数据可靠性提高；不能改为连续 DMA 模式

### 4. 队列 + 信号量的双重同步
- **设计决策**: TX 任务使用队列存储待发送帧，信号量等待 DMA 完成
- **原因**: 队列提供缓冲，信号量提供硬件同步，避免帧重叠或丢失
- **约束**: 初始化时必须先 `osSemaphoreRelease()` 一次，表示硬件初始状态空闲
- **影响**: 不能简化为单纯的队列或单纯的信号量，两者缺一不可

### 5. FreeRTOS 堆大小的严格限制
- **设计决策**: 堆大小仅 15360 字节（约 15KB）
- **原因**: STM32F407VET6 的 192KB RAM 中需要预留空间给其他用途
- **约束**: 所有任务栈、队列、信号量都从这个堆中分配，新任务需谨慎计算
- **影响**: 不能随意添加大栈的任务，系统可扩展性受限

### 6. 电机参数的宏定义集中管理
- **设计决策**: 所有电机参数（轮径、轮距、编码器脉冲数）集中在 `bsp_motor.h` 中定义为宏
- **原因**: 便于硬件更换时统一修改，避免参数散落在代码各处
- **约束**: 更换电机或轮子时必须同时修改这些宏，否则速度计算完全错误
- **影响**: 不能在运行时动态修改这些参数（除非通过协议 ID `0x08` 下发 PID 参数）

### 7. CubeMX 代码保护的强制约束
- **设计决策**: 所有业务逻辑必须在 `/* USER CODE BEGIN/END */` 块内
- **原因**: CubeMX 重新生成时会保留这些块，但删除块外的代码
- **约束**: 不能在 CubeMX 生成的代码中直接修改，必须通过 CubeMX 界面修改配置
- **影响**: 每次重新生成后必须检查链接脚本和源文件列表，否则编译失败

### 8. CMake 构建系统的文件显式注册
- **设计决策**: 新增 `.c` 文件必须在 `CMakeLists.txt` 中显式添加
- **原因**: CMake 不会自动扫描目录，需要显式指定源文件
- **约束**: 不能依赖隐式的文件发现机制
- **影响**: 新增文件时容易遗漏，导致编译时文件被忽略

### 9. printf 重定向的阻塞模式
- **设计决策**: `_write()` 函数使用 `HAL_UART_Transmit()` 阻塞模式
- **原因**: 确保 printf 输出完整，避免数据丢失
- **约束**: 不能改为 DMA 非阻塞模式，否则输出可能不完整
- **影响**: printf 调用会阻塞任务，不适合高频调用

### 10. 分层架构的严格分离
- **设计决策**: 应用层、硬件抽象层、底层驱动严格分离
- **原因**: 便于维护和移植，降低耦合度
- **约束**: 应用层不能直接调用 HAL 库函数，必须通过 BSP 层
- **影响**: 新增功能时必须遵守分层原则，否则代码耦合度增加

### 11. 协议 ID 的扩展性限制
- **设计决策**: 协议 ID 采用 8-bit 编码，最多支持 256 种消息类型
- **原因**: 简化协议设计，减少帧头大小
- **约束**: 新增协议 ID 时必须遵守 6 字节数据区的限制
- **影响**: 如果需要传输大量数据，必须采用分包机制或修改协议格式

### 12. 任务延时的强制约束
- **设计决策**: 禁止在任何任务中使用 `HAL_Delay()`，必须使用 `vTaskDelay()` 或 `osDelay()`
- **原因**: `HAL_Delay()` 会阻塞整个 FreeRTOS 内核，导致其他任务无法运行
- **约束**: 这是硬性约束，不能违反
- **影响**: 系统响应性和实时性完全依赖于这个约束的遵守
